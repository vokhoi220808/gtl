Trusted Verify Registry

QR/hash tự mang theo chỉ chứng minh file khớp với thông tin trong link. Registry mới giúp biết bản ghi đó có được đăng ký thật không, có bị thu hồi không, có hết hạn không, và file PDF hiện tại có bị sửa so với bản gốc không.

Mở app PDF FusionMở trang Verify

✅ THẬT

Verify ID có trong registry, trạng thái active, chưa hết hạn, và SHA-256 của PDF tải lên khớp bản ghi gốc.

❌ GIẢ / ĐÃ SỬA

Verify ID có trong registry nhưng SHA-256 của PDF tải lên khác bản gốc. Chỉ cần sửa 1 byte là hash sẽ khác.

⚠️ CHƯA XÁC ĐỊNH

ID hoặc hash chỉ nằm trong link/certificate nhưng chưa được registry công khai ghi nhận. Không nên gọi là bản thật.

🚫 THU HỒI / HẾT HẠN

Bản ghi tồn tại nhưng đã bị admin thu hồi hoặc quá hạn xác minh. Trang verify sẽ cảnh báo rõ.

Cách hoạt động

  1. App xuất PDF và tính SHA-256 cho file kết quả.
  2. Admin đăng ký Verify ID + SHA-256 + metadata vào data/verify-registry.json qua API hoặc commit thủ công.
  3. Khi người dùng mở verify.html, trang kiểm tra link/certificate và gọi /api/verify.
  4. Người dùng upload PDF cần kiểm tra. Trình duyệt tính SHA-256 tại chỗ, rồi so với registry.

Cấu hình backend registry trên Vercel

Trong Vercel → Project Settings → Environment Variables, thêm:

VERIFY_ADMIN_SECRET=mat_khau_admin_manh
PFSP_VERIFY_SIGNING_SECRET=chuoi_ky_hmac_rieng
GITHUB_TOKEN=github_pat_co_quyen_contents_write
GITHUB_OWNER=ten_owner
GITHUB_REPO=ten_repo
GITHUB_BRANCH=main
PFSP_REGISTRY_PATH=data/verify-registry.json

Nếu chưa cấu hình GitHub token, API vẫn kiểm tra được registry tĩnh, nhưng khi đăng ký sẽ trả về JSON patch để bạn tự commit vào data/verify-registry.json.

Endpoint

GET /api/verify?id=PFSP-...&sha256=...&size=...
POST /api/verify { "action":"register", "certificate": {...}, "adminSecret":"..." }
POST /api/verify { "action":"revoke", "id":"PFSP-...", "reason":"...", "adminSecret":"..." }
